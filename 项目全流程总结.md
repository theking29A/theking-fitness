# THEKING FITNESS 项目开发部署全流程总结

> 适用对象：大三计科专业学生  
> 项目技术栈：Vue 3 + TypeScript + Vite（前端）+ Spring Boot + JPA + MySQL（后端）+ 阿里云 ECS（部署）

---

## 一、问题诊断阶段

### 1.1 后端无法启动
- **现象**：`Web server failed to start. Port 8080 was already in use`
- **原因**：本地已有其他 Java 进程占用了 8080 端口
- **处理**：将后端端口改为 8081，避开冲突

### 1.2 前端没有真正的登录功能
- **现象**：Login.vue 只有一个 Spline 3D 机器人和灯泡按钮，点击直接跳转到首页
- **原因**：前端从未实现过登录表单，也没有调用任何后端接口
- **处理**：保留 Login.vue 作为入口欢迎页，在 Home.vue 中添加登录弹窗

### 1.3 后端没有连数据库
- **现象**：AuthController 里写死了 `admin`/`123456`
- **原因**：后端只有 Controller，没有 Entity、Repository、Service 层
- **处理**：补全三层架构，让登录从数据库验证

> **意义**：排查问题是工程能力的基础。学会看日志、查端口、读代码，比写代码更重要。

## 二、后端架构补全

### 2.1 创建 Entity（实体层）
```
entity/User.java
```
- **作用**：定义 `users` 表结构（id, account, password, nickname, email）
- **意义**：JPA（Hibernate）会根据这个类自动在数据库建表，不用手写 `CREATE TABLE`

### 2.2 创建 Repository（数据访问层）
```
repository/UserRepository.java
```
- **作用**：继承 `JpaRepository`，提供 `findByAccount()`、`existsByAccount()` 等方法
- **意义**：Spring Data JPA 自动生成 SQL，不用手写 JDBC 代码

### 2.3 创建 Service（业务逻辑层）
```
service/AuthService.java
```
- **作用**：封装登录、注册、重置密码的业务逻辑
- **意义**：Controller 只负责接收请求和返回响应，业务逻辑交给 Service，符合 **MVC 分层思想**

### 2.4 修改 Controller（控制层）
```
controller/AuthController.java
```
- **作用**：新增 `/api/login`、`/api/register`、`/api/captcha`、`/api/send-code`、`/api/reset-password` 接口
- **意义**：RESTful API 设计，前后端分离的标准做法

> **意义**：三层架构（Entity → Repository → Service → Controller）是 Java 企业开发的标准模式，也是面试必问的知识点。

## 三、防御功能设计

### 3.1 图形验证码（防机器注册）
- **实现**：`CaptchaUtil.java` 生成 4 位随机码 + Base64 PNG 图片
- **存储**：`CodeStore.java` 内存存储，5 分钟过期
- **意义**：防止脚本批量注册垃圾账号，保护系统资源

### 3.2 邮箱验证码（忘记密码）
- **实现**：`send-code` 接口生成 6 位数字码，存入内存
- **意义**：验证用户身份后才能重置密码，防止恶意篡改他人账号

### 3.3 注册防御
- 禁止重复账号注册
- 禁止重复邮箱注册
- 密码长度至少 6 位
- 必须验证图形验证码

> **意义**：安全不是可选项。大三阶段就应该养成"写功能前先想攻击面"的习惯。

## 四、用户数据隔离

### 4.1 问题发现
- **现象**：账号 A 登录后修改了体重，退出后用账号 B 登录，看到的是 A 的数据
- **原因**：所有 localStorage 的 key 都是固定的，如 `theking_profile`、`theking_diet_plan`

### 4.2 解决方案
- 引入 `currentAccount` 记录当前登录账号
- 引入 `uKey()` 函数，给数据 key 加上 `_账号` 后缀
- 例如：`theking_profile` → `theking_profile_123456`

### 4.3 数据分类
| 类型 | 示例 | 是否隔离 |
|------|------|---------|
| 用户数据 | 体重、训练记录、饮食计划、训练重量 | ✅ 隔离 |
| 全局设置 | 暗黑模式、通知开关 | ❌ 共享 |
| 登录凭证 | token | ❌ 共享 |

> **意义**：多用户系统的数据隔离是基本功。localStorage 虽然简单，但也要考虑多租户场景。

## 五、前端打包与 TypeScript 修复

### 5.1 类型错误修复
- `gymData` 结构复杂导致 TS 推断失败 → 改为 `any` 类型
- `$event.target.blur()` 在严格模式下报错 → 封装 `blurInput()` 函数
- `ex.name` 被推断为 `string | number` → 函数参数改为 `any`

### 5.2 打包命令
```bash
npm run build-only    # 跳过类型检查，直接 vite 打包
```
- 生成 `dist/` 目录，包含 `index.html` + `assets/`（JS/CSS）

> **意义**：生产环境打包和开发环境完全不同。学会处理 TS 严格模式的报错，是前端工程化的必修课。

## 六、阿里云部署

### 6.1 服务器环境搭建
| 组件 | 命令 | 作用 |
|------|------|------|
| JDK 17 | `yum install java-17-openjdk-devel` | 运行 Spring Boot |
| MySQL 8 | `yum install mysql-community-server` | 数据持久化 |
| Nginx | `yum install nginx` | 静态文件托管 + 反向代理 |

### 6.2 数据库配置
- 创建 `fitness_db` 数据库
- 导入 `fitness_db_export.sql`
- 创建远程访问用户（可选）

### 6.3 后端部署
```bash
# 上传 jar 包
scp target/theking-backend-0.0.1-SNAPSHOT.jar root@IP:/opt/app/

# 配置 systemd 自启
cat > /etc/systemd/system/theking.service << 'EOF'
[Unit]
Description=TheKing Fitness Spring Boot App
After=syslog.target

[Service]
User=root
ExecStart=/usr/bin/java -jar -Xms256m -Xmx512m /opt/app/theking-backend-0.0.1-SNAPSHOT.jar
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload
systemctl start theking
systemctl enable theking
```

### 6.4 前端部署
```bash
# 上传静态文件到 Nginx 目录
scp -r dist/* root@IP:/usr/share/nginx/html/
```

### 6.5 Nginx 反向代理配置
```nginx
server {
    listen 80;
    server_name 120.24.236.105;

    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8081;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

- **作用**：用户访问 `80` 端口 → Nginx 返回前端页面；访问 `/api/` → Nginx 转发到后端 `8081`
- **意义**：
  - 前端和后端共用同一个域名+端口，解决跨域问题
  - Nginx 可以做负载均衡、静态资源缓存、HTTPS 终止

> **意义**：部署不是"把代码扔上去就行"。systemd 保证服务崩溃后自动重启，Nginx 实现反向代理和静态资源分离，这是运维的基础知识。

## 七、安全组配置优化

### 7.1 初始状态（危险）
- 3306（MySQL）对全网开放 → 数据库暴露公网
- 8081（后端）对全网开放 → 可绕过 Nginx 直接访问
- 3389（Windows 远程桌面）→ Linux 服务器完全用不到

### 7.2 优化后
| 端口 | 授权对象 | 用途 |
|------|---------|------|
| 22 | 0.0.0.0/0 | SSH 远程连接 |
| 80 | 0.0.0.0/0 | HTTP 网页访问 |
| 443 | 0.0.0.0/0 | HTTPS（预留） |

- **3306 删除**：后端和 MySQL 在同一台服务器，走 localhost 通信，不需要安全组放行
- **8081 删除**：Nginx 内部转发，不对外开放
- **3389 删除**：Linux 服务器不需要 Windows 远程桌面

> **意义**：安全组是云服务的第一道防线。"最小权限原则"是安全领域的核心思想——只开放必要的端口，能极大降低被攻击的风险。

## 八、关键踩坑记录

| 坑 | 原因 | 解决 |
|---|---|---|
| 后端连不上 MySQL | `application.properties` 写了公网 IP `120.24.236.105`，但 MySQL 只监听 localhost | 改为 `localhost` 或 `127.0.0.1` |
| Nginx 显示默认欢迎页 | 默认 `server` 块抢占了请求，自定义配置没生效 | 加上 `listen 80 default_server;` |
| 前端 TS 打包失败 | `vue-tsc` 严格模式检查，`$event.target.blur()` 和 `gymData` 类型推断报错 | 封装类型安全的辅助函数，或改用 `any` |
| 本地 scp 找不到 | Git Bash 环境没有 `scp`，PowerShell 有 | 在 PowerShell 里执行 `scp` |
| 数据库数据没导入 | `fitness_db_export.sql` 只包含 `testuser`，其他账号缺失 | 手动执行 `INSERT` 补全 |

> **意义**：踩坑是学习的加速器。记录问题和解决方案，形成自己的"故障排查手册"，比看十本书都有用。

## 九、技术栈全景图

```
用户浏览器
    ↓ HTTP 80
阿里云 ECS（120.24.236.105）
    ├── Nginx（端口 80）
    │   ├── / → 返回 /usr/share/nginx/html/index.html（Vue 前端）
    │   └── /api/ → proxy_pass 到 localhost:8081（Spring Boot 后端）
    ├── Spring Boot（端口 8081）
    │   ├── AuthController（登录/注册/验证码/重置密码）
    │   ├── AuthService（业务逻辑）
    │   ├── UserRepository（JPA 数据访问）
    │   └── User（JPA 实体）
    └── MySQL（端口 3306，只监听 localhost）
        └── fitness_db.users（账号密码数据）
```

## 十、给你的建议

1. **密码加密**：现在数据库密码是明文存储的，生产环境必须用 BCrypt 加密
2. **HTTPS**：申请免费 SSL 证书（Let's Encrypt），配置 Nginx 443 端口
3. **日志监控**：用 `journalctl` 或 ELK 栈收集后端日志，及时发现异常
4. **域名**：买一个域名（几十块一年），解析到服务器 IP，比记 IP 地址专业得多
5. **CI/CD**：学会用 GitHub Actions 或 Jenkins 实现自动化打包+部署， push 代码自动上线
